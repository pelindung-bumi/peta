This guide covers how to operate and maintain the Pelindung Bumi blog at [pelindungbumi.dev](https://pelindungbumi.dev).

## Repository Overview

The blog lives at [pelindung-bumi.github.io](https://github.com/pelindung-bumi/pelindung-bumi.github.io) and is built with:

- **Astro**: Framework for the site
- **Bearnie**: Base template for design
- **Bun**: Package manager and script runner
- **Tailwind CSS**: Styling via Vite plugin

### Project Structure

```text
src/
├── components/          # Reusable Astro components
├── content/
│   ├── blog/           # Blog posts (MDX files)
│   └── docs/           # Internal docs (if any)
├── layouts/            # Page layouts (BlogPost.astro, etc.)
├── pages/              # Route pages
├── styles/             # Global CSS
└── utils/              # Utility functions
public/                 # Static assets
scripts/                # Build scripts (generate-og.mjs)
```

## Development Workflow

All commands use Bun. Run from the project root:

```bash
# Install dependencies
bun install

# Start local development server
bun run dev
# Site: http://localhost:4321

# Build production site
bun run build

# Preview production build
bun run preview

# Run arbitrary Astro commands
bun run astro ...
```

## Creating Blog Posts

### Step-by-Step

1. Create a new `.mdx` file in `src/content/blog/`
2. Use kebab-case for the filename (e.g., `my-new-post.mdx`)
3. Add frontmatter at the top of the file

### Frontmatter Fields

```yaml
title: 'My Post Title'
description: 'A brief description for SEO and social sharing'
tags: ['kubernetes', 'deployment', 'best-practices']
pubDate: 2026-03-17
writerGithub: 'your-github-username'
updatedDate: 2026-03-17  # Optional: when updating an old post
heroImage: ./hero.jpg     # Optional: featured image in public/
```

### Post Template

```mdx
---
title: 'Understanding Kubernetes Deployments'
description: 'What a Deployment actually does and why we use it.'
tags: ['kubernetes', 'deployments']
pubDate: 2026-03-17
writerGithub: 'saktil'
---

import CodeExample from '../../components/CodeExample.astro';

Your introduction paragraph here.

## Your First Section

<CodeExample
  filename="example.yaml"
  lang="yaml"
  code={`apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3`}/>

Your explanation of the code.

## Another Section

More content here...
```

### Using Code Examples

The blog has a `CodeExample` component for consistent code blocks:

```mdx
<CodeExample
  filename="deployment.yaml"
  lang="yaml"
  code={`apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3`}/>
```

Supported languages: `yaml`, `bash`, `typescript`, `javascript`, `go`, `python`, etc.

### Writing Guidelines

- Write as curious people learning in public, not as experts
- Keep content practical and code-focused
- Use clear, direct explanations
- Include working examples
- Assume readers are learning infrastructure, Kubernetes, cloud, SRE, or DevOps

## OG Image Generation

The blog generates Open Graph images for social sharing automatically using a custom SVG template.

### Running the Generator

```bash
bun run generate:og
```

This creates `public/social/pelindung-bumi-og.png` (1200x630).

### How It Works

The `scripts/generate-og.mjs` script:

1. Creates an SVG template with:
   - Pelindung Bumi branding and colors
   - Site title and tagline
   - Logo placement
   - Subtle gradients and patterns for visual interest

2. Composites the logo using `sharp`:
   - Resizes logo to fit the template
   - Places it in the top-right corner
   - Uses the forest green color scheme

3. Outputs a PNG ready for social media

### Customizing OG Images

Edit `scripts/generate-og.mjs` to change:

- **Colors**: Change hex values in the gradient and text definitions
- **Text**: Update the headline, subtext, and tagline
- **Logo**: Change `LOGO_W`, `LOGO_H`, and positioning variables
- **Layout**: Modify SVG elements (rectangles, text positioning)

### Color Palette

The OG image uses the blog's dark mode colors:

```text
Background:     #091310 (deep forest green)
Foreground:     #EDF1E8 (off-white light)
Primary:        #DCE6D3 (soft green accent)
Muted fg:       #9FB0A7 (gray-green)
Subtle glow:    #1A3D2F (lighter forest green)
```

## RSS Feed & Sitemap

The blog automatically generates both:

### RSS Feed

- **Location**: `/rss.xml`
- **Generated by**: `@astrojs/rss`
- **Configuration**: `src/pages/rss.xml.js`

No manual configuration needed. New posts automatically appear in the feed.

### Sitemap

- **Location**: `/sitemap-index.xml`
- **Generated by**: `@astrojs/sitemap`
- **Configuration**: `astro.config.mjs`

The sitemap includes all blog posts, pages, and follows Google's guidelines.

### Adding to Feed Readers

Use this URL in any RSS reader:

```text
https://pelindungbumi.dev/rss.xml
```

## Git Metadata for Writer Info

The blog automatically tracks who wrote and modified posts using Git history.

### How It Works

The `writerGithub` field in frontmatter manually sets the author. The `utils/blogGitMeta` utility:

1. Reads the Git log for each blog post
2. Extracts the GitHub usernames of people who:
   - Authored commits for the file
   - Modified the file over time

### Displaying Writer Info

The blog post layout (`src/layouts/BlogPost.astro`) shows:

- **Writer**: The `writerGithub` value from frontmatter
- **Modifiers**: List of GitHub usernames who contributed via commits

This metadata links to GitHub profiles for verification.

### Local Git Names

Local git names (e.g., "Your Name") are **not** shown publicly. The utility filters to only show GitHub usernames from commits.

### Updating Writer Info

If a new person writes a post:

1. Set `writerGithub: 'their-username'` in frontmatter
2. Git history will automatically capture modifications

## Deployment to GitHub Pages

The blog deploys automatically to GitHub Pages:

- **Domain**: [pelindungbumi.dev](https://pelindungbumi.dev)
- **Trigger**: Push to `main` branch
- **Workflow**: `.github/workflows/deploy.yml`

### Deployment Workflow

1. Checkout the repository
2. Run `bun install` and `bun run build` (via withastro/action@v5)
3. Upload the `dist/` folder to GitHub Pages
4. The site is live within 1-2 minutes

### Manual Deployment

You can also trigger a deployment manually from GitHub Actions.

### Viewing Deployments

Go to: `https://github.com/pelindung-bumi/pelindung-bumi.github.io/actions`

## Content Management

### Updating an Existing Post

1. Edit the `.mdx` file in `src/content/blog/`
2. Update `updatedDate` in frontmatter (if relevant)
3. Commit and push

### Deleting a Post

1. Remove the `.mdx` file
2. Commit and push (GitHub will 404 links, which is fine)

### Organizing Posts

Posts are sorted by `pubDate` by default. The blog index shows them in reverse chronological order.

### Tags

Tags are used for categorization. Current tags used:

- `kubernetes`
- `deployment`
- `scaling`
- `cloud`
- `devops`
- `sre`
- `nixos`
- `infrastructure`

### Hero Images

Optional hero images go in `public/blog/` and reference them:

```yaml
heroImage: ./blog/kubernetes-hero.jpg
```

Images should be:
- **Size**: At least 1200px wide
- **Format**: JPG, PNG, or WebP
- **Optimize**: Under 500KB if possible

## Common Tasks

### Preview a Post Without Publishing

Create the file locally. It will show up in your dev server at `/blog/your-slug/`. It won't be deployed until you commit to `main`.

### Check If a Post Renders Correctly

Run:

```bash
bun run build
```

If there are no errors, the post is valid.

### Fix a Broken Build

If GitHub Actions fails:

1. Check the error in the Actions tab
2. Common issues:
   - Syntax error in MDX (unclosed tag, bad YAML)
   - Missing import for a component
   - Invalid frontmatter value
3. Fix and push again

### Rebuild Without New Content

Make a whitespace-only change or update the `package.json` version, then push.

## Troubleshooting

### Dev Server Won't Start

```bash
# Clear install and reinstall
rm -rf node_modules bun.lockb
bun install
bun run dev
```

### OG Image Not Updating

```bash
# Regenerate manually
bun run generate:og

# Clear browser cache when testing
# Shift + Cmd + R (Chrome/Edge)
```

### RSS Feed Empty

1. Check that posts have `pubDate` set
2. Verify `src/content/blog/` has `.mdx` files
3. Clear dev server cache: stop and restart `bun run dev`
